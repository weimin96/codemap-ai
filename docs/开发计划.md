# 项目快速接管工作台：最终版开发计划

> 项目代号：Project Fast Onboarding Workbench  
> CLI 名称：`pfo`  
> 目标形态：通过 npm 安装，命令行指定本地项目目录启动，在浏览器中打开本地代码理解工作台；配置 AI 后可分析项目、生成项目地图、预览代码、追踪链路、绑定上下文追问，并逐步沉淀为可复用的项目接管知识库。

---

## 1. 项目背景

当前核心场景是：公司项目经过大量 AI 修改后，负责人或接手人一开始并不了解项目如何运作，需要快速建立第一版认知。传统方式是自己从目录、README、入口文件、路由、Service、Model 一层层看，速度慢且容易迷失。

本项目的目标不是让 AI “替人吃透项目”，而是让 AI 先快速生成一张可靠度可控的项目地图，再让使用者通过代码预览、链路验证、断点建议和上下文追问，在最短时间内完成本人接管。

核心原则：

```text
AI 先给地图，人再沿地图验证。
先抓主干，再补细节。
每个结论必须绑定代码证据、文件位置、可信度和验证方式。
```

## 2. 产品定位

### 2.1 一句话定位

一个本地运行的 AI 代码项目快速接管工作台。

### 2.2 目标用户

- 刚接手陌生项目的开发负责人。
- 需要快速理解 AI 大量修改后代码的人。
- 需要审查 AI 生成代码是否破坏业务逻辑的人。
- 团队内负责项目交接、新人 onboarding、代码审计的人。
- 独立开发者维护多个历史项目时快速恢复上下文。

### 2.3 核心使用流程

```text
npm install -g project-fast-onboarding
pfo /path/to/project

浏览器打开本地工作台
  ↓
扫描项目
  ↓
配置 AI Provider
  ↓
生成项目地图、入口列表、核心链路、风险雷达
  ↓
点击链路/文件预览代码
  ↓
围绕当前文件、函数、链路、选中代码追问
  ↓
把确认后的结论沉淀为项目接管报告
```

### 2.4 项目边界

本工具不是普通代码补全工具，也不是单纯文档生成器。它的核心竞争点是：

- 面向“本人快速接管项目”。
- 强调核心链路、入口、数据副作用和风险。
- 把 AI 解释和代码预览绑定。
- 把追问绑定到文件、函数、链路、选中代码。
- 每条结论标注事实、推测、不确定点。
- 支持本地运行和私有代码场景。

## 5. 技术栈规划

### 5.1 当前推荐技术栈

| 层 | 技术 |
|---|---|
| 运行环境 | Node.js 20+ |
| CLI | Commander |
| 本地服务 | Express 或 Fastify |
| 前端框架 | Vite + React + TypeScript |
| 样式 | Tailwind CSS |
| UI 组件 | shadcn/ui |
| 图标 | lucide-react |
| 代码预览 | Monaco Editor |
| 图表 | Mermaid |
| AI SDK | Vercel AI SDK |
| OpenAI | `@ai-sdk/openai` |
| OpenAI-compatible | `@ai-sdk/openai-compatible` |
| Ollama | AI SDK Ollama Provider 或兼容 Provider |
| 搜索 | ripgrep |
| 符号解析 | Tree-sitter |
| 本地存储 | SQLite |
| 向量检索 | sqlite-vec / LanceDB / Chroma 三选一 |
| 配置 | `pfo.config.json` + 环境变量 |
| 打包 | npm package |
| 日志 | pino |
| 测试 | Vitest + Playwright |
| 代码质量 | ESLint + Prettier |
| 发布 | GitHub Actions + npm publish |

### 5.2 为什么这样选

- npm 安装和 CLI 启动要求决定运行时必须以 Node.js 为主。
- React + Vite 适合快速构建本地 Web 工作台。
- Tailwind + shadcn/ui 能快速做出可维护 UI。
- Monaco 是代码预览和选中代码追问的关键。
- Mermaid 适合 AI 直接生成图表 DSL。
- Vercel AI SDK 避免自己封装各家模型 API。
- Tree-sitter 负责跨语言符号索引。
- ripgrep 负责快速文本搜索。
- SQLite 负责本地持久化，不引入服务器数据库。
- 后期向量检索可作为可选增强，不影响基础功能。

## 8. Prompt / Skill 设计

### 8.1 系统角色

```text
你是项目快速接管助手。你的目标不是泛泛解释代码，而是帮助用户本人快速接管陌生项目。

你必须：
1. 先建立项目地图。
2. 再提取入口、模块、核心链路和风险。
3. 每个结论标注：确定事实 / 合理推测 / 不确定点。
4. 每个结论尽量绑定具体文件、函数、行号。
5. 优先回答“我下一步应该看什么”。
6. 不确定时必须说明需要验证哪里。
7. 面向快速接管，而不是完整教学。
```

### 8.2 项目分析输出要求

AI 输出必须是 JSON，不直接输出自由 Markdown，便于 UI 渲染：

```json
{
  "overview": {},
  "entrypoints": [],
  "modules": [],
  "flows": [],
  "risks": [],
  "readingPlan": [],
  "unknowns": [],
  "diagrams": []
}
```

### 8.3 追问上下文包

每次追问只带相关上下文：

```text
当前项目概要：
当前文件：
当前符号：
当前链路：
当前风险：
当前选中代码：
相关调用方：
相关被调用方：
用户问题：
```

### 8.4 快捷问题

文件级：

- 解释这个文件。
- 这个文件在项目里承担什么职责？
- 主流程函数是哪几个？
- 哪些细节可以先跳过？
- 这个文件和哪些模块有关？

函数级：

- 解释这个函数。
- 输入输出是什么？
- 谁调用它？
- 它调用谁？
- 有什么副作用？
- 有什么异常路径？
- 改这里影响哪里？

链路级：

- 生成代码剧本。
- 找数据读写。
- 找状态变化。
- 推荐断点。
- 检查链路是否完整。
- 找可能的 AI 修改风险。

风险级：

- 为什么危险？
- 怎么验证？
- 需要加什么测试？
- 如果修复，影响范围是什么？

## v0.3：文件索引与 Tree-sitter 符号索引

目标：

- 让工具从“文件级理解”升级为“函数/类/符号级理解”。

参考：

- Aider Repo Map。
- RepoMapper。

功能：

1. 引入 Tree-sitter。

6. 点击符号定位到代码行。

- 集成 `tree-sitter` 或 `web-tree-sitter`。
- 维护语言 parser 映射。
- 设计 fallback：Tree-sitter 失败时用正则提取。
- 增加索引缓存。

验收标准：

- 打开文件时能展示函数/类列表。
- 点击函数能跳转到对应行。
- AI 追问中能带上当前函数代码。
- 大型项目扫描不会明显卡死。

## v0.4：Repo Map 与上下文压缩

目标：

- 让 AI 不依赖全量代码，而是基于压缩后的 repo map 建立项目初始认知。

参考：

- Aider Repo Map：重要类和函数的紧凑地图。
- Repomix：AI 友好的仓库打包格式。

功能：

1. 生成 `repo-map.json`。
2. 按文件优先级、符号数量、入口概率、调用密度排序。
3. 生成适合 AI 的 `context-pack`。
4. 支持 token budget。
5. 分析时优先投喂：
   - README
   - package/config
   - entrypoints
   - route files
   - service files
   - model/schema
   - repo map
6. UI 显示“AI 本次分析使用了哪些文件”。

技术任务：

- 实现 `ContextPackBuilder`。
- 实现 token 预算估算。
- 实现 ignore 规则：`.gitignore`、自定义 `pfo.ignore`。
- 支持导出 `project-context.md`。

验收标准：

- 中型项目可以在一次分析中生成稳定项目地图。
- AI 报告不再依赖随机扫描顺序。
- 每个 AI 结论能追溯到文件来源。

目标：

- 从普通项目总结升级为“核心链路剧本”。

参考：

- DeepWiki 的图表化项目理解。
- CodeWiki 的仓库级文档结构。
- Mermaid 图表能力。

功能：

## v0.6：本地 RAG 与代码追问增强

目标：

- 让追问不再只依赖当前文件，而是可以自动检索相关代码。

参考：

- Cody / Sourcegraph 的代码库问答。
- RAGFlow 的上下文层。
- OpenDocuments 的混合检索思路。
- Continue 的上下文绑定聊天。

功能：

1. 本地全文搜索。
2. 代码块切片。
3. 语义索引可选开启。
4. 混合检索：
   - keyword
   - symbol
   - path
   - embedding
5. 回答时附带引用：
   - 文件路径
   - 行号范围
   - 相关代码片段

## 19. 最终成功标准

项目达到完整版时，用户应能在 30-120 分钟内完成一个陌生项目的第一层接管：

```text
知道项目怎么启动。
知道主要入口在哪里。
知道核心模块是什么。
知道最重要的 2-5 条业务链路。
知道数据在哪里读写。
知道哪些地方风险最高。
知道下一步应该看哪些文件。
能围绕当前代码继续追问。
能导出一份项目接管报告。
```

本项目的最终价值不是替用户“看懂项目”，而是显著缩短从陌生代码库到可维护状态的时间。
